# 负荷预测与容载比反推助手 - 开发文档

## 1. 架构目标

本项目采用桌面端架构：

```text
PySide6 UI
↓
core 业务模型
↓
Excel 读取/写回
```

核心原则：

```text
UI 层可以调用 core
core 层不能依赖 PySide6
```

这样做的原因：

- 后续可以把 UI 从 PySide6 替换为 Web 或命令行；
- 公式/预测/规则逻辑可以独立测试；
- 业务模型不会被界面代码污染。

## 3. 核心模块说明

### 3.1 workbook_loader.py

职责：

- 读取 `.xlsx`；
- 读取 `.xls`；
- 必要时生成安全修复副本；
- 输出统一的 LoadedWorkbook。

为什么单独做：

```text
Excel 文件格式兼容问题很多，读取逻辑必须和业务计算隔离。
```

### 3.2 template_parser.py

职责：

- 识别 sheet 角色；
- 动态识别年份列；
- 识别项目库；
- 抽取地市/区县容载比表和校核表中的核心指标；
- 输出 MetricRecord 长表。

为什么用长表：

```text
area | voltage | metric | year | value
```

比直接操作 Excel 单元格更稳定，也更适合新增年份和多目标求解。

### 3.3 forecast_engine.py

职责：

- 根据容载比目标反推网供负荷；
- 根据增长率目标生成平滑预测；
- 将地市负荷变化分摊到区县；
- 检查同时率风险；
- 生成三类方案。

当前方案：

1. 优先调整负荷；
2. 负荷容量均衡；
3. 区外送受电兜底。

为什么不直接复刻 Excel 公式：

```text
模板中大量关系没有显式写成公式，例如地市—区县关系、同时率、业务锁定、项目投产口径。
这些更适合做成业务模型，而不是做完整 Excel 计算器。
```

### 3.4 rule_engine.py

职责：

- 读取默认规则；
- 校验容载比；
- 计算同时率可调范围。

默认业务口径：

- 容载比：1.3–2.5；
- 配变负载率：0–1，软目标 0.5；
- 同时率：0.83–0.99，最大调整 0.1；
- 2025 年及以前锁定。

### 3.5 rule_persistence.py

职责：

- 保存规则档案；
- 自动命名规则文件；
- 构造工作簿结构签名；
- 加载时校验规则和模板是否匹配；
- 对目标预设和规则项做逐条校验。

为什么要结构签名：

```text
同名模板可能改变了 sheet、年份列或关键指标，如果盲目套用旧规则，可能误导反推结果。
```

### 3.6 workbook_writer.py

职责：

- 复制原始工作簿；
- 在副本中写入调整值；
- 必要时扩展新增年份列；
- 新增“预测结果表”；
- 输出写回日志。

重要原则：

```text
原始传入文件绝不修改。
```

## 4. 数据模型

核心数据类在 `core/models.py` 中。

### 4.1 MetricRecord

表示一条业务指标记录：

```text
区域、电压等级、指标、年份、值、来源 sheet、来源单元格
```

为什么保留来源：

```text
反推结果要写回 Excel 副本，必须知道原始数据来自哪张表、哪一行、哪一列。
```

### 4.2 ForecastTarget

表示用户设置的目标：

- 容载比；
- 增长率；
- 负荷；
- 配变平均负载率。

### 4.3 ForecastRules

表示全局规则边界。

### 4.4 Adjustment

表示一条建议调整。

字段包括：

```text
调整对象、区域、电压等级、指标、年份、原值、新值、原因、风险、来源单元格
```

### 4.5 ScenarioResult

表示一套完整反推方案。

## 5. 方法注释规范

第二版开始，新增或修改的方法应包含“是什么、为什么”说明。

推荐格式：

```python
def some_function(...):
    """是什么：说明这个方法做什么。

    为什么：说明为什么需要这个方法，以及它对应的业务或架构原因。
    """
```

要求：

1. 不只写“处理数据”这类空泛描述；
2. 要说明业务原因，例如“为了支持2031新增年份写回”；
3. UI 方法要说明界面动作和业务意图；
4. core 方法要说明计算口径和约束原因。

## 6. 规则档案结构

规则文件保存在：

```text
rules/{模板主名称}_规则.yaml
```

主要结构：

```yaml
meta:
  rule_name: xxx_规则
  workbook_base_name: xxx
  source_workbook_name: xxx.xlsx
  workbook_signature:
    structure_hash: ...

rules:
  model:
    latest_actual_year: 2025
    forecast_start_year: 2026
  ratio_rules:
    hard_min: 1.3
    hard_max: 2.5

target_presets:
  - target_type: 容载比
    area: 内江市
    voltage_level: 110kV
    year: 2030
    target_value: 1.85

rule_items: []
```

## 7. 写回策略

写回流程：

```text
复制原始文件
↓
按调整项定位 sheet / 行 / 年份列
↓
如果年份列不存在，插入新列并复制样式
↓
按公式覆盖策略写入值
↓
新增预测结果表
↓
保存新文件
```

公式覆盖策略：

- 允许覆盖公式：写入结果值；
- 不允许覆盖公式：跳过该单元格，并记录到写回清单。

## 8. 测试

运行：

```bash
pytest
```

当前测试覆盖：

- 动态项目库识别；
- 动态年份识别；
- 容载比目标反推负荷；
- 导出副本不修改原文件；
- 预测结果表生成。

后续建议增加：

- 规则档案匹配测试；
- 失效目标预设校验测试；
- 2031+ 年份扩展写回测试；
- 同时率越界预警测试；
- 区外送受电兜底测试。

## 9. 开发注意事项

1. 不要在 core 中导入 PySide6。
2. 不要直接修改原始 Excel 文件。
3. 不要把年份写死到 2030。
4. 不要把项目库 sheet 名写死。
5. 新增业务口径应优先写入 `ForecastRules` 或配置文件。
6. 写回逻辑必须记录日志，方便业务人员审查。
7. 任何涉及现状年的调整必须默认禁止。
8. 区外送受电只能作为低优先级兜底变量，不能静默使用。
9. 新增方法必须写“是什么、为什么”注释。
